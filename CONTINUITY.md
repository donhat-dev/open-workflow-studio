## Key Decisions:
- **Lucide font trade-off**: `lucide-static` is viable for a no-JS icon path, but it ships the full font and global `.icon-*` class mapping. If adopted here, prefer vendored local assets over CDN and use a namespaced wrapper/helper strategy to reduce collisions with existing CSS class conventions.
- **Persistent project memory**: Add `memory-bank/` with the standard Cline core files (`projectbrief.md`, `productContext.md`, `activeContext.md`, `systemPatterns.md`, `techContext.md`, `progress.md`) plus project rules instructing agents to read/update it. `CONTINUITY.md` stays as a temporary bridge for existing Copilot-specific workflows.
- **Expression contract**: Strict `=` prefix for expressions. Legacy bare `{{ }}` fallback removed from both frontend and backend.
- **Custom nodes**: `x_` prefix = custom (`is_custom`). Runtime code in `workflow.type.code` only, executed via `safe_eval`, permission via `workflow.type.group_id`.
- **Node-type architecture**: Backend-driven only. Odoo registry populated dynamically from `workflow.type` RPC â†’ dynamic node factory â†’ registry refresh. No static JS node files, no fallback.
- **Schema mapping**: Backend `config_schema`/`input_schema`/`output_schema` â†’ frontend `Control`/`Socket` via shared mapper with fallback defaults.
- **Socket types**: `DataSocket` (default), `ErrorSocket` (explicit error routing), `TriggerSocket` (reserved future).
- **`_input` context**: Dict payloads merge top-level keys with metadata keys (`json`, `item`, `items`). Same shape in backend + frontend preview.
- **UI layout**: Odoo App Card style nodes (Icon Left, Title/Subtitle Right). History panel docked left. Execution inspector docked bottom (List/Gantt/Raw JSON).
- **Command/Hotkey**: PdfManager pattern â€” `useActiveElement` scoping, `useCommand` for discoverable actions, `useHotkey` for navigation. Custom keyboard hook kept behind feature flag.
- **model_select**: Fixed-only, no expression toggle, literal text preserved as-is.
- **Trigger architecture**: Hybrid approach (ADR-008). Trigger nodes in canvas (UX), Odoo infra for activation (ir.cron, base.automation, webhook). `env.context` passed as trigger runner output. `base_automation` optional dependency. Webhook response mode configurable (immediate/last_node). Schedule cron uses `state='code'`. Changed fields only for record events.
- **Connector architecture direction (Accepted, 2026-04-20)**: ADR-010 now separates workflow organization from provider configuration. `workflow.workspace` is the primary organizational boundary for `ir.workflow`, while new model `workflow.connector` owns provider-specific config, endpoint/auth presets, and `workflow.http.request.connector_id` bindings. `connector_request` stays a distinct managed node type; snapshot-first bridge records remain the runtime contract.
- **Production-safe / business-safe execution direction (Proposed, 2026-04-19)**: Draft ADR-013 proposes a PostgreSQL-backed, queue-first resilience architecture for `workflow_studio` that borrows n8n-style admission/worker separation and Camunda-style lease/retry/incident semantics while explicitly accepting Odoo/Python limits. Directionally: `debug` vs `production_safe` vs `business_safe` run modes, durable run leases, node-boundary checkpoints, batch cursor resume, side-effect intent ledger for idempotency, and operator-facing incidents instead of silent orphaned runs. Deterministic replay / exactly-once external effects remain out of scope.
